# OpenSpec (SDD) — Wishlot

Спек-драйвен девелопмент для проекта Wishlot.

## Структура

```
openspec/
├── specs/                    # Долгоживущие спецификации по доменам
│   ├── wishlist/
│   │   └── spec.md           # Управление желаниями
│   ├── wheel/
│   │   └── spec.md           # Колесо фортуны и выбор
│   ├── ui/
│   │   └── spec.md           # UI-поведение и темы
│   └── ...
└── changes/                  # Активные изменения
    ├── <change-name>/        # Папка изменения
    │   ├── proposal.md       # Зачем и что (intent + scope)
    │   ├── design.md         # Как (technical approach)
    │   ├── tasks.md          # Чеклист реализации
    │   └── specs/            # Дельта-спеки — что меняется
    │       └── <domain>/
    │           └── spec.md
    └── archive/              # Завершённые изменения
        └── <change-name>/
            └── ...
```

## Артефакты изменения

| Файл | Отвечает на |
|------|-------------|
| `proposal.md` | Зачем? Что в scope, что out of scope |
| `design.md` | Как? Архитектурные решения, data flow |
| `tasks.md` | Шаги. Нумерованный чеклист с чекбоксами |
| `specs/` | Что меняется в долгоживущих спеках |

## Спеки

Спек — это **контракт поведения**, не план реализации.

- `## Purpose` — что описывает этот спек
- `### Requirement:` — конкретное поведение системы (SHALL / MUST / SHOULD)
- `#### Scenario:` — Given / When / Then

Избегай в спеках: имён классов, библиотек, пошаговой реализации.

## Workflow

1. **Предложить** — создать папку в `changes/`, написать `proposal.md`
2. **Специфицировать** — написать дельта-спеки в `changes/<name>/specs/`
3. **Спроектировать** — `design.md` с техрешениями
4. **Реализовать** — `tasks.md` + код
5. **Заархивировать** — перенести папку в `changes/archive/`, обновить `specs/`
