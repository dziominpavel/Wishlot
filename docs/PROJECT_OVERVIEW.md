# Wishlot — обзор продукта

> Дата: 2026-05-18 · Статус: **спецификация**

## Идея

**Wishlot** помогает не сорваться на импульсивные покупки и при этом **радовать себя осознанно**. Пользователь ведёт список желаемых покупок с ориентировочной стоимостью. Когда наступает момент «могу потратить до N рублей на себя», приложение:

1. Отбирает только те желания, которые **влезают в бюджет** и ещё **не исполнены**.
2. **Случайно** выбирает одно (чтобы снять мучительный выбор «а что взять?»).
3. Показывает результат с **игровой анимацией** (колесо фортуны).
4. Просит **согласие**: взял — помечаем исполненным; отказался — желание остаётся в списке.

Философия:

- **Осознанность** — бюджет задаётся явно перед «розыгрышем».
- **Радость без вины** — случайность снимает паралич выбора.
- **Контроль** — без автоматической покупки; только отметка в приложении.
- **Offline-first** — данные только на устройстве в MVP.

## Аудитория

- Пользователи Android 8+, которые ведут список «хочу купить» и периодически выделяют себе небольшой бюджет на радость.
- MVP: **русский** интерфейс; валюта — символ из локали устройства (₽ по умолчанию для ru).

## Основные сценарии

### S1. Добавить желание в вишлист

1. Экран **Вишлист** → «Добавить».
2. Поля: **название** (обязательно), **цена** (обязательно, > 0), заметка (опционально).
3. Сохранение → статус `ACTIVE`, сортировка в списке по дате добавления или цене (настраиваемо позже).

### S2. Редактировать / удалить желание

1. Тап по карточке → детали.
2. Изменить название, цену, заметку.
3. Удалить (с подтверждением) или архивировать (фаза 2+).

### S3. «Побаловать себя» — розыгрыш

1. Экран **Побаловать себя** → пользователь вводит **бюджет** («могу потратить до …»).
2. Приложение применяет **фильтр**: `price <= budget` и `status = ACTIVE`.
3. Если кандидатов **0** — понятный empty state («нет желаний в этом бюджете») + ссылка на вишлист.
4. Если кандидатов **≥ 1** — анимация **колеса** (сегменты = кандидаты или визуальная имитация при большом N).
5. Остановка на **одном** случайно выбранном желании (алгоритм — [WISH_PICK_LOGIC.md](WISH_PICK_LOGIC.md)).
6. Карточка результата: название, цена, заметка.
7. Кнопки: **«Беру!»** / **«Не сейчас»**.

### S4. Согласие — исполнено

1. «Беру!» → статус желания `FULFILLED`, `fulfilledAt = now`.
2. Запись в **Историю** исполненных.
3. Snackbar / короткая анимация успеха.
4. Желание **исчезает** из активного вишлиста и из будущих розыгрышей.

### S5. Отказ

1. «Не сейчас» → желание остаётся `ACTIVE`.
2. Возврат на экран ввода бюджета или предложение **крутить снова** (тот же бюджет).
3. Повторный розыгрыш может снова выпасть то же желание — это ожидаемо (равномерный random).

### S6. История

- Вкладка **История**: исполненные желания, сортировка `fulfilledAt DESC`.
- Просмотр без редактирования цены (или только заметка — на обсуждение).

### S7. Настройки

- Символ валюты / формат суммы (DataStore).
- (Фаза 2+) Сортировка вишлиста по умолчанию.
- (Фаза 3+) Экспорт JSON.

## Модель данных

### Wish (желание)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Long PK | |
| `clientId` | String UUID | Для бэкапа |
| `title` | String | Название («Наушники Sony») |
| `priceMinor` | Long | Цена в **минорных единицах** (копейки), избегаем float |
| `note` | String? | Заметка |
| `status` | Enum | `ACTIVE`, `FULFILLED`, `ARCHIVED` |
| `createdAt` | Long | Epoch millis |
| `fulfilledAt` | Long? | Когда отметили исполненным |
| `sortOrder` | Int? | Ручной порядок (фаза 2) |

### PickSession (опционально, фаза 2+)

Лог сессий розыгрыша для аналитики «сколько раз крутил» — **не в MVP**.

### AppSettings (DataStore)

| Ключ | Описание |
|------|----------|
| `currencyCode` | ISO 4217, default из Locale |
| `lastTreatBudgetMinor` | Последний введённый бюджет (удобство UX) |
| `wheelAnimationEnabled` | bool, default true |
| `confirmBeforeFulfill` | bool, default true (всегда показывать кнопки) |

## Статусы желания

| Статус | В вишлисте | В розыгрыше | В истории |
|--------|------------|-------------|-----------|
| `ACTIVE` | ✅ | ✅ | ❌ |
| `FULFILLED` | ❌ | ❌ | ✅ |
| `ARCHIVED` | ❌ (или отдельный раздел) | ❌ | опционально |

MVP: достаточно `ACTIVE` и `FULFILLED`. `ARCHIVED` — отложить.

## Предложения: что в MVP / что позже

### MVP (фазы 0–3)

| Функция | Зачем |
|---------|--------|
| CRUD вишлиста | Ядро продукта |
| Фильтр по бюджету | Бизнес-правило |
| Случайный выбор + колесо | УТП приложения |
| Подтверждение / отказ | Не помечать без согласия |
| История исполненных | Ощущение прогресса |
| Room + offline | Без аккаунта |

### Позже

| Идея | Фаза |
|------|------|
| Категории (техника, книги, …) | 4 |
| Приоритет / вес в random | 4 |
| Статистика («потратил бы N за год») | 5 |
| Виджет «крутить» | 6 |
| Облако / синхронизация | вне scope MVP |
| Ссылки на маркетплейсы | 4 |

### Не делать в первой версии

| Идея | Почему |
|------|--------|
| Оплата / корзина магазина | Другое приложение |
| Совместные вишлисты | Усложняет модель |
| Push «пора побаловать» | Нужны разрешения и контент-стратегия |
| ML-рекомендации | Не нужны для random pick |

---

## Вне scope

- Интеграция с банками и реальными тратами.
- Подарки другим людям (только личный вишлист).
- Мультивалютные желения в одном списке (MVP — одна валюта из настроек).

## Нефункциональные требования

| Требование | Цель |
|------------|------|
| Min SDK | **26** (Android 8.0+) |
| Target / compile SDK | **36** (как GymProgress / VoiceMind) |
| Хранение цен | `Long` minor units, форматирование в UI |
| Производительность | 500+ желаний без лагов списка (LazyColumn) |
| Приватность | Только локальная БД; без аналитики с PII в MVP |
| Доступность | contentDescription на иконках; крупные кнопки на экране результата |
