# Руководство: внедрение Spec-Driven Development (SDD) на любом проекте

> **Временный файл.** Описывает пошагово, как настроить процесс spec-driven development на новом проекте — pet-проект, промышленный сервис, любой стек.
> После настройки можно удалить или перенести в вики.

---

## Что такое SDD (в нашем понимании)

**Спецификация — источник истины.** Любое изменение начинается с документации, а не с кода. AI (Cursor / Windsurf / Copilot) сначала правит спеку, ждёт апрува человека, только потом пишет код и тесты.

**Ключевые принципы:**
- Сначала спека, потом код, потом тесты.
- Каждая доработка — через промежуточный `pending`-файл для согласования.
- Версионирование спек: Git + явные пометки в тексте.
- Cross-repo зависимости описываются в спеке и реализуются после основного кода.

---

## Шаг 1. Определить источники истины

В любом проекте есть документация, которая описывает поведение системы. Это может быть:

| Тип проекта | Примеры источников истины |
|-------------|---------------------------|
| **Backend (Java/Node/Go)** | `docs/api.md`, `docs/entities.md`, Swagger/OpenAPI, `docs/architecture.md` |
| **Frontend (React/Angular/Vue)** | `docs/components.md`, `docs/routing.md`, `docs/state-management.md` |
| **Report/Analytics** | `docs/reports/<report>.md`, `docs/metrics.md` |
| **DSL/Scheduler** | `docs/dsl-operators.md`, `docs/scheduler.md` |
| **Mobile** | `docs/screens.md`, `docs/api-integration.md` |

**Правило:** если документация размазана по README, комментариям и чатам — собрать её в `docs/` в виде markdown-файлов. Один файл = одна логическая область.

---

## Шаг 2. Структура папок (минимум)

Создать в корне проекта:

```
project/
  docs/
    README.md              # оглавление + описание SDD-процесса
    changes/
      TEMPLATE.md          # шаблон pending-изменения
      pending/
        .gitkeep           # чтобы папка попала в Git
    <feature>.md           # источники истины по зонам
    architecture.md        # общая архитектура (если нужно)
  .windsurf/
    rules/
      spec-driven.mdc      # авто-правило для Windsurf
  .cursor/
    rules/
      spec-driven.mdc      # авто-правило для Cursor
```

**Если проект не Java / не Maven:** путь `docs/` может быть `specs/`, `docs/`, `.docs/` — главное, чтобы AI знал путь из правил.

---

## Шаг 3. Создать правила для AI

### Для Windsurf (Cascade)

Файл `.windsurf/rules/spec-driven.mdc`:

```yaml
---
description: Spec-Driven Development workflow — pending-файлы, согласование, cross-repo
alwaysApply: true
---

# Spec-Driven Development (SDD)

## Обязательный порядок работы на КАЖДУЮ сессию

1. **СНАЧАЛА СПЕЦИФИКАЦИЯ**
   - Если изменение касается существующей фичи — сначала правим/дополняем спеку в `docs/`.
   - Новые фичи — сначала создаём полную спеку в `docs/<feature>.md`.
   - Никогда не пиши код до того, как спека согласована.

2. **PENDING-ФАЙЛ ДЛЯ СОГЛАСОВАНИЯ**
   - Создай `docs/changes/pending/<feature>-<краткое-описание>.md` по шаблону `docs/changes/TEMPLATE.md`.
   - Этот файл — единственное, что видит пользователь на этапе согласования. Никакого кода.
   - Жди апрува от пользователя перед следующим шагом.

3. **ПОСЛЕ АПРУВА — ПРИМЕНИТЬ К ИСТОЧНИКУ ИСТИНЫ**
   - Внеси изменения из pending-файла в основную спеку (`docs/*.md`).
   - Удали pending-файл — он больше не нужен.

4. **ПОТОМ КОД**
   - Пиши/правь код **строго** в соответствии со спекой.
   - Если спека и код расходятся — спека приоритетнее; либо вернись к шагу 1.

5. **ТЕСТЫ**
   - Обнови/дополни тесты под новое поведение из спеки.
   - Запусти тесты. Убедись, что проходят.
   - Если тест не проходит — либо баг в коде, либо спека уточняется (вернуться к шагу 1).

6. **CROSS-REPO ЗАВИСИМОСТИ**
   - Если спека содержит раздел «Зависимости от других сервисов»:
     - Сначала реализуй всё в текущем репо.
     - Затем переходи к зависимым репозиториям с отдельными pending-файлами.
     - Ссылки на cross-repo изменения фиксируй в спеке.

## Правила форматирования pending-файла

- Заголовок: `# [PENDING] <Feature> — <краткое описание>`
- Разделы обязательны: `## Изменения`, `## Мотивация`, `## Зависимости от других сервисов` (если есть), `## Проверка / Критерии приёмки`
- Для каждого изменения указывай: `ADDED`, `MODIFIED` или `REMOVED`.
- После апрува pending-файл удаляется; история хранится в Git.

## Что НЕЛЬЗЯ делать

- Править код без обновления спеки.
- Создавать pending-файл и сразу писать код — жди апрува.
- Оставлять pending-файлы после завершения работы.
```

### Для Cursor

Файл `.cursor/rules/spec-driven.mdc` — **тот же текст**, но с `globs` в frontmatter:

```yaml
---
description: Spec-Driven Development workflow — pending-файлы, согласование, cross-repo
globs: ["docs/**/*.md", "src/**/*.{java,ts,tsx,py,go}"]
---
```

(В `globs` указать расширения файлов вашего проекта.)

## Шаг 4. Создать шаблон pending-изменения

Файл `docs/changes/TEMPLATE.md`:

```markdown
# [PENDING] <Feature> — <краткое описание изменения>

> **Статус:** `PENDING` — ожидает согласования пользователя
> **Фича:** `<Feature>`
> **Дата создания:** YYYY-MM-DD

---

## Мотивация

## Изменения

### ADDED
- Что добавляется.
- Ссылка на раздел спеки, который будет дополнен.

### MODIFIED
- Что изменяется.
- Было: `...`
- Станет: `...`
- Ссылка на раздел спеки, который будет изменён.

### REMOVED
- Что удаляется (если применимо).

## Зависимости от других сервисов

- [ ] `<repo>` — что нужно.

## Проверка / Критерии приёмки

- [ ] Unit-тесты обновлены/добавлены.
- [ ] Интеграционные тесты проходят.
- [ ] Ручная проверка: ...

---

> **Инструкция для AI:** После согласования пользователя:
> 1. Примени изменения из этого файла к основной спеке `docs/*.md`.
> 2. Удали этот pending-файл.
> 3. Реализуй код в соответствии с обновлённой спекой.
> 4. Обнови тесты. Запусти. Убедись, что проходят.
```

## Шаг 5. Обновить docs/README.md

Добавить в начало:

```markdown
## 🎯 Процесс: Spec-Driven Development (SDD)

Спецификация — **источник истины** для кода. Каждое изменение проходит через этапы:

1. **Спецификация** — AI правит/дополняет спеку (`docs/`).
2. **Согласование** — создаётся `docs/changes/pending/<feature>-<описание>.md`, пользователь читает и апрувит.
3. **Применение** — изменения вносятся в основную спеку, pending-файл удаляется.
4. **Код** — AI пишет код строго по спеке.
5. **Тесты** — AI обновляет тесты и проверяет прохождение.
6. **Cross-repo** — если в спеке указаны зависимости от других репо, реализуются после основного кода.

> **Правило:** код правится **только после** спеки. Никакого кода до апрува.
```

## Шаг 6. Добавить в .gitignore (если нужно)

Если `.cursor/` и `.windsurf/` не должны попадать в Git:

```gitignore
### AI IDE ###
.cursor/
.windsurf/
```

(Правила для AI локальные, их не нужно коммитить. А `docs/changes/` — часть репо, должно быть в Git.)

---

## Шаг 7. Первое тестовое изменение

Чтобы отработать процесс, выбрать безопасную доработку:
- Добавить в спеку недостающий раздел (например, «граничные случаи» или «формат чисел»).
- Или: уточнить существующее описание, поправив дубли.

**Порядок:**
1. AI создаёт `docs/changes/pending/<feature>-<описание>.md`.
2. Человек читает и говорит «апрув».
3. AI применяет изменения в основную спеку, удаляет pending.
4. AI пишет/проверяет код.
5. AI обновляет тесты, запускает, проверяет прохождение.

---

## Шаг 8. Распространение на другие репозитории

Если проект состоит из нескольких сервисов (backend, frontend, report, stat):

1. Повторить шаги 1–6 в каждом репо.
2. В правилах `spec-driven.mdc` указать имена связанных репозиториев.
3. В `pending`-файлах всегда включать раздел «Зависимости от других сервисов».
4. Cross-repo изменения: сначала текущий репо, потом зависимые — с отдельными pending-файлами.

- [ ] `.windsurf/rules/spec-driven.mdc` создан.
- [ ] `.cursor/rules/spec-driven.mdc` создан.
- [ ] `docs/README.md` содержит описание SDD-процесса.
- [ ] Первое изменение отработано по всем 6 шагам.
- [ ] Пользователь уверен, что AI сначала правит спеку, а не код.

## Резюме

**Всё, что нужно для старта:**
1. Папка `docs/` с источниками истины.
2. Два файла правил: `.windsurf/rules/spec-driven.mdc` и `.cursor/rules/spec-driven.mdc`.
3. Шаблон `docs/changes/TEMPLATE.md`.
4. Описание процесса в `docs/README.md`.
5. Одно тестовое изменение для отработки.

После этого любая доработка будет идти: **спека → согласование → код → тесты**.
